# 说说页 / 树洞页列表加载与分页方案

## 1. 结论

当前两页“无限向下滚动但没有显式分页”的设计，**不应视为同一类问题**。

### 1.1 说说页（`SayPage.vue`）

**部分合理，但当前还不闭环。**

原因：

- 说说页本质上是“近期动态流”，天然适合沉浸式连续浏览；
- 当前页面视觉语言也是明显的 feed / timeline 语义：`SayPageHero`、连续卡片流、轻提示哨兵文案、发布浮层都在强化“刷流”体验；
- 但当前只有 `IntersectionObserver + loadMore`，缺少：
  - 位置感知
  - 总量感知
  - 跳转能力
  - 恢复能力

这会导致：

- 用户知道自己“还可以往下刷”，但不知道“已经刷到哪里”；
- 刷到很深时，无法快速回到较早位置；
- 刷新后只能从第一页重新进入，不利于继续阅读。

**结论：说说页可以保留无限滚动，但不应只有无限滚动。**

### 1.2 树洞页（`TreeholePage.vue`）

**当前不合理。**

原因：

- 树洞页不是纯 feed，它同时承担：
  - 公共匿名内容流
  - 可展开评论讨论
  - 弹幕入口页
- 这种内容更接近“可回看、可定位、可跳转”的讨论列表，而不是纯刷流产品；
- 现状里 `useTreeholePage.ts` 只请求 `page: 1, pageSize: 50`，页面列表层没有任何“下一页”或“页码”的 UI，等于：
  - 旧内容不可达
  - 用户无从判断是否还有更早树洞
  - 产品在信息架构上形成“伪无限流”错觉，但实际上只是加载一页

**结论：树洞页应改为显式分页主导，不应伪装成无限流。**

---

## 2. 当前实现概况

### 2.1 说说页

文件：

- `src/modules/say/pages/SayPage.vue`

当前行为：

- 本体列表使用：
  - `page = ref(1)`
  - `pageSize = 20`
  - `hasMore`
  - `IntersectionObserver`
- 通过底部 `loadMoreSentinel` 命中后自动追加下一页；
- 页面底部只有：
  - “向下滚动加载更多”
  - “已经到底了”
- 没有显式分页器，没有页码状态，没有快速跳转。

这是一套标准的“内容流追加模型”，但 UI 只做了“继续向下”的单向引导，没有做“我现在在哪里”的状态反馈。

### 2.2 树洞页

文件：

- `src/modules/treehole/pages/TreeholePage.vue`
- `src/modules/treehole/composables/useTreeholePage.ts`

当前行为：

- 页面主列表在 `loadTreeholes()` 中只拉：
  - `getTreeholes({ page: 1, pageSize: 50 })`
- 页面模板没有列表分页器；
- 评论区内部倒是有自己的分页：
  - `CommentThread`
  - `SPagination`

说明当前产品在“树洞主列表”和“树洞评论列表”上采用了两套完全不同的信息导航策略：

- 评论：显式分页
- 主列表：无分页、且无继续加载

这是不一致的。

## 3. 结合产品设计语言的判断

本产品前台的视觉特征不是传统资讯站，而是：

- 强情绪背景
- 玻璃 / 半透明分层容器
- Hero 驱动
- 卡片化内容流
- 轻操作芯片、quiet 状态反馈

因此分页不应该做成“后台表格式”的重控制器，但也不能完全没有导航。

### 3.1 说说页适合的分页语言

说说页应维持：

- 连续流
- 轻提示
- 安静的状态反馈

不适合：

- 列表顶部和底部都塞一整排强存在感页码按钮
- 强管理后台式“第 1 2 3 4 5 页”密集控制

### 3.2 树洞页适合的分页语言

树洞页更适合：

- 明确的批次边界
- 可回到指定页
- 评论展开后能保持上下文

因为树洞不是单纯消费流，而是“内容 + 讨论”的双层结构。对这种页面，过于流式会损伤可检索性和可回访性。

## 4. Naive UI 是否有现成方案

当前项目使用版本：

- `naive-ui@^2.44.1`

核对结果：

### 4.1 有：`NPagination`

当前项目已经通过 `SPagination` 封装接入 `NPagination`。

可直接利用的能力包括：

- `simple`
- `page`
- `pageSize`
- `itemCount`
- `showQuickJumper`
- `showSizePicker`
- `prefix`
- `suffix`

适合做：

- 树洞主列表分页
- 说说页的“轻分页补充导航”

### 4.2 没有：真正的 `NInfiniteScroll`

当前版本里没有现成的 `NInfiniteScroll` 组件可直接接住页面级无限流。

这意味着：

- 无限滚动仍然要靠：
  - `IntersectionObserver`
  - 或自定义滚动监听 composable
- Naive UI 无法直接替你解决“自动加载 + 页码同步 + 恢复位置”的组合问题。

### 4.3 不推荐把 `NScrollbar` 当解决方案

理论上可以用容器滚动配合 `NScrollbar` 或滚动事件做局部无限列表。
但对当前产品不推荐，原因：

- 会制造内层滚动容器，破坏整页滚动语义；
- 与当前 `PageShell / glass-page / hero + section` 的页面语言冲突；
- 对评论展开、浏览器历史位置恢复、移动端惯性滚动都不友好。

**结论：Naive UI 可以很好地承接分页 UI，但无限滚动仍应由项目自身状态模型管理。**

---

## 5. 建议方案

### 5.1 总策略

采用 **分页面分策略**，不要强行统一：

- 说说页：**无限滚动主导 + 轻分页补充**
- 树洞页：**显式分页主导**

这是最符合当前产品语义和信息结构的方案。

---

## 6. 说说页方案：无限滚动主导 + 轻分页补充

### 6.1 目标

保留“刷动态”的自然感，但补上位置感、总量感和可恢复性。

### 6.2 建议交互

#### 第一层：保留现有无限滚动

保留：

- `IntersectionObserver`
- 自动加载下一页
- 底部 quiet 提示

因为它符合说说页的 feed 气质。

#### 第二层：增加轻量分页状态条

在列表底部哨兵上方或 hero 下方增加一条轻量状态带，内容例如：

- 已加载 `40 / 128` 条
- 当前在第 `2 / 7` 页
- 最近更新：最新动态

表现形式建议：

- `SSurfaceChip variant="counter"`
- 配合轻文案，不做重分页条

这样用户能知道：

- 现在不是“无穷无尽”地刷
- 当前已处于哪一页批次

#### 第三层：增加“查看更早动态”按钮

在自动滚动加载 2~3 个批次后，切换成**手动加载更多**更合理。

建议策略：

- 前 2 页：自动加载
- 第 3 页起：显示 quiet button
- 文案：
  - “继续查看更早动态”
  - “加载第 4 页”

原因：

- 长内容流不应无限自动追加到底；
- 手动确认更符合“浏览旧动态”的语义；
- 可以避免页面过长、性能和定位一起恶化。

#### 第四层：可选提供 `simple` 分页器

在桌面端底部增加一个 `SPagination simple` 作为补充导航，不作为主控件。

作用：

- 快速回到上一批 / 下一批
- 让用户知道列表本质仍是分页数据

但不建议默认使用完整页码阵列。

### 6.3 状态模型建议

说说页列表层应从“只会 append”的模型，升级为：

- `page`
- `pageSize`
- `total`
- `loadedPages`
- `autoLoadUntilPage`
- `mode: 'auto' | 'manual'`

这样才能支撑：

- 前几页自动滚动
- 后续手动加载
- 轻分页展示
- 刷新恢复当前页批次

### 6.4 视觉建议

保持当前 say 的语言：

- 不用重型页码条
- 用 `SSurfaceChip` / quiet button / 细分隔线
- 用“更早动态”“已加载 x/x”这类 feed 语义文案

不要引入后台式表格分页观感。

---

## 7. 树洞页方案：显式分页主导

### 7.1 目标

让树洞主列表具备：

- 可达旧内容
- 可定位
- 可恢复
- 与评论区分页策略一致

### 7.2 建议交互

#### 第一层：主列表改为真正分页

`useTreeholePage.ts` 不再固定只拉 `page: 1, pageSize: 50`，而是改成：

- `page`
- `pageSize`
- `total`
- `hasMore` 可选，但不是主控状态

#### 第二层：在树洞流底部使用 `SPagination`

直接使用现有 `SPagination`，桌面端推荐：

- 标准模式
- 显示总数摘要
- 支持前后页切换

移动端推荐：

- `simple`
- 只保留上一页 / 下一页 + 当前页信息

#### 第三层：页码切换后滚回树洞流顶部

这是必须项。

因为树洞页顶部还有一个满屏 hero + barrage，用户不能在翻到第 5 页后仍被留在页面底部不知所措。

建议定位到：

- `.messages-section` 顶部
- 而不是整页最顶端 hero

#### 第四层：评论展开状态与页码联动

页码切换时：

- 自动关闭当前展开的树洞评论
- 或至少清空 `activeTreeholeId`
- 避免翻页后展开态指向已不存在的楼层

### 7.3 视觉建议

树洞页可以比说说页更明确地展示分页，因为它本来就更像“讨论列表”。

但仍应保持当前产品语言：

- 分页器放在 section 内底部
- 配合 `SSurfacePanel` 的柔和分隔
- 避免厚重背景条
- 文案建议：
  - “第 2 / 8 页”
  - “共 156 条树洞”

不要做成后台管理页的 dense pager。

---

## 8. 最终推荐

### 8.1 说说页

**推荐采用：混合加载方案**

- 保留无限滚动作为主体验
- 增加轻状态反馈
- 2~3 页后切换为手动加载更多
- 可选补 `SPagination simple` 作为弱导航

### 8.2 树洞页

**推荐采用：显式分页方案**

- 主列表接入 `SPagination`
- 页面级分页状态进入 `useTreeholePage`
- 切页回到 `messages-section` 顶部
- 切页清理展开评论上下文

---

## 9. 为什么这是最符合当前产品设计语言的

因为它同时满足三件事：

1. **说说页保持 feed 感**
- 不把动态流硬做成后台分页列表

2. **树洞页恢复信息架构完整性**
- 不让旧树洞不可达
- 不让页面只有第一页假装是无限流

3. **继续复用现有 UI 抽象层**
- 用 `SPagination`
- 用 `SSurfaceChip`
- 用现有 quiet / glass / panel 语言
- 不引入新的重型 third-party feed 组件

---

## 10. 实施优先级

### P0

- 树洞主列表接入真正分页
- 页面底部加 `SPagination`
- 切页回到 `messages-section` 顶部

### P1

- 说说页增加 `已加载 x / 总数` 与 `当前页` 状态反馈
- 说说页从“纯自动追加”升级为“前几页自动、后续手动”

### P2

- 说说页补 `simple` 分页弱导航
- 支持根据 URL / 历史状态恢复到最近浏览批次

---

## 11. 一句话结论

- **说说页：无限滚动可以保留，但必须补上弱分页与状态闭环。**
- **树洞页：不应该继续无分页，应该改成显式分页主导。**
- **Naive UI 能解决分页 UI，但不能替代你自己的列表加载状态机。**
