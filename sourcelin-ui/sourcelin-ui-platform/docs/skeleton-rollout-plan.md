# 全站 Skeleton 方案综合评判与实施计划

本文用于评估当前「统一使用 `SSkeleton` 作为加载占位」方案的可用性，并给出后续可执行的最优落地路径。

## 1. 综合评判（高级前端视角）

### 1.1 方案可用性结论

结论：**可用且方向正确**，建议继续推进为全站标准。

原因：

- 已满足架构边界：业务层统一使用 [`SSkeleton.vue`](../src/shared/components/ui/SSkeleton.vue)，不直接使用 `n-skeleton`，符合 UI 抽象层约束。
- 已在文章域验证：标签、分类、归档、正文、评论、发文页等核心场景已完成替换，`typecheck` 与架构测试可通过。
- 体验收益明确：相比纯 `spinner`，骨架更能表达布局预期，降低“空白等待”感。

### 1.2 当前方案风险与不足

- **复用颗粒度不均**：文章域已有 `ContextArticleCardSkeleton`，但首页、用户中心等仍有重复样式点。
- **首屏与增量加载策略未完全分层**：部分场景适合骨架（首屏），部分场景更适合轻提示（load more）。
- **文档缺统一约束**：何时用骨架、何时保留 spinner、何时复用组件，尚未形成全局规则。

---

## 2. 最优方案（推荐标准）

### 2.1 设计原则

1. **首屏结构化加载优先骨架**  
   只要是“卡片/列表/面板”首屏，优先 `SSkeleton`。

2. **增量加载保持轻量**  
   已有内容在屏上时（如 infinite load more），默认使用轻文案/小 spinner，不强制大骨架，避免跳动。

3. **组件级复用优先于页面内拼样式**  
   同形态占位抽成组件（如文章卡、评论行、统计卡），页面只组合，不重复定义骨架结构。

4. **视觉与主题统一**  
   Skeleton 颜色、圆角、间距全部走现有变量体系，禁止业务层硬编码颜色。

### 2.2 分层策略

| 层级 | 职责 | 推荐做法 |
|------|------|---------|
| 页面壳（路由级） | 应用初始化、全局遮罩 | 保留 `StartupLoadingOverlay` 一类能力 |
| 区块级（页面内） | 结构化首屏加载 | 使用 `SSkeleton` 或复用骨架组件 |
| 列表增量加载 | load more 状态 | 默认轻提示（quiet / spinner） |
| 操作级加载 | 提交、点赞、删除等 | 使用按钮 `loading` 或局部转圈 |

---

## 3. 场景清单与优先级

### P0（优先立即做）

1. **首页 `page-container-wrap` 内首屏加载统一骨架**
   - `HomeArticlesSection`：首屏空列表时改为文章卡骨架。
   - `WaterfallList`：提供默认骨架插槽（或由业务传入统一骨架）。
   - `HomeSiteStats`：统计卡区域由 spinner 改为 skeleton grid。

2. **导航页首屏加载可视化**
   - `NavigationPage` 在 `loading` 且无数据时，增加导航卡片骨架，避免“中间空白”。

### P1（第二阶段）

1. **用户中心四个列表**
   - `UserArticleList` / `UserCollectList` / `UserFollowList` / `UserFanList`：`SSpin` 改为列表骨架。

2. **Treehole / SearchDialog**
   - 树洞列表、搜索结果首屏占位由 spinner 切换为对应骨架。

### P2（规范化与治理）

1. 抽象通用骨架组件库（建议目录）
   - `src/shared/components/feedback/skeletons/`
   - 例如：`SkeletonArticleCard`、`SkeletonStatGrid`、`SkeletonNavCard`、`SkeletonCommentRow`

2. 规则文档补齐
   - 在规则文档中新增“加载态规范”章节（何时骨架、何时 spinner、组件命名规则、验收标准）。

---

## 4. 关键决策（针对首页 `page-container-wrap`）

### 结论

**不建议**给 `page-container-wrap` 容器本身包一层统一骨架。  
**建议**在其内部业务区块分别做骨架（统计、文章、瀑布流）。

### 理由

- `page-container-wrap` 是布局容器，不承载单一数据请求，做整体骨架会弱化真实结构信息。
- 分区骨架更贴近真实内容形态，感知质量更高，后续维护成本更低。
- 不同区块请求完成时机不同，分区渲染可渐进展示，避免“一刀切等全部完成”。

---

## 5. 实施步骤（建议）

1. **定义通用骨架规范**（半天）
   - 命名、尺寸 token、边距、过渡、`aria-busy` 约定。

2. **完成 P0 三块改造**（1~2 天）
   - 首页三块 + 导航页。

3. **回归测试**
   - 桌面端/移动端、深浅色主题、慢网模拟（Fast 3G）。

4. **完成 P1 列表改造**（1~2 天）
   - 用户中心 + Treehole + SearchDialog。

5. **沉淀文档与组件清单**（半天）

---

## 6. 验收标准

### 体验验收

- 首屏等待不出现“纯空白内容区”。
- 骨架结构与真实结构一致（卡片/行/网格布局匹配）。
- 增量加载不出现明显布局抖动。

### 工程验收

- 业务层不直接使用 `naive-ui` 的 `NSkeleton`。
- 无新增硬编码颜色，全部走现有设计变量。
- `npm run typecheck` 与 `npm run test:architecture` 通过。

### 可访问性验收

- 加载区设置 `aria-busy="true"`。
- 骨架元素加 `aria-hidden="true"`（或放在容器语义下避免噪声朗读）。

---

## 7. 建议的最终实施结论

当前方案**可直接作为全站标准继续推进**。  
最优路径是：**先做首页与导航页（P0），再做用户中心等列表（P1），最后沉淀成骨架规范与组件库（P2）**。  
该路径风险低、收益高，且与现有分层架构和设计系统保持一致。
