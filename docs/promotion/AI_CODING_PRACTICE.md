# AI 协作开发实践

本文档用于对外说明 Sourcelin Blog 如何将 AI 编程能力沉淀为仓库级公开资产，而不是停留在单次对话或单个 IDE 插件配置中。

如果你正在评估一个开源项目是否真正适合 AI 协作开发，核心不在于“有没有用过某个模型”，而在于仓库是否已经把边界、规则、验证和发布流程写清楚。

## 这份实践适合谁

- 想把现有项目整理成可长期协作的 AI Coding 仓库
- 想让 Codex、Cursor、OpenCode、Claude Code、Qoder、Trae 等工具在同一套约束下工作
- 想把博客系统、毕设项目或二开项目整理成对外可信的开源仓库

## 本项目如何使用 AI

Sourcelin Blog 当前这轮开源整理、README 重写、文档结构收口、公开边界治理和多远端发布流程，均由 Codex 完成。

但项目本身并不依赖某个单独工具。真正可复用的是以下仓库资产：

- [AGENTS.md](../../AGENTS.md)：定义执行入口、目录边界、开源分支边界和发布模型
- [rules/README.md](../../rules/README.md)：统一 API 契约、编码规范和验证要求
- [skills/README.md](../../skills/README.md)：把规则变成按任务执行的工作流

这意味着只要 AI 编程工具能读取仓库文件、遵守规则并基于代码执行，它就可以接入这套体系。

## 为什么它兼容多种 AI 编程工具

本项目支持的重点不是“某个工具的品牌名”，而是“仓库约束已经文件化”：

- 支持 Codex
- 支持 Cursor
- 支持 OpenCode
- 支持 Claude Code
- 支持 Qoder
- 支持 Trae
- 也兼容通义灵码、MarsCode、GitHub Copilot 及其他支持读取仓库规则的工具

原因很简单：

1. 任务入口写在仓库里，不写在聊天窗口里
2. API 契约和分页规则写在仓库里，不靠人记忆
3. 验证命令写在仓库里，不靠“应该没问题”
4. 开源分支与内部资料边界写在仓库里，不靠临时口头约定

## 本项目沉淀了哪些可复用做法

### 1. 把规则写进仓库，而不是写进一次性提示词

很多项目的问题不是没用 AI，而是每次都从头解释一遍上下文。这样做的结果是：

- 工具一换，规则丢了
- 会话一断，边界丢了
- 人一多，风格乱了

Sourcelin Blog 的做法是把约束拆成稳定文件：

- `AGENTS.md`
- `rules/`
- `skills/`

这样仓库本身就是长期上下文。

### 2. 把“可公开”和“仅内部使用”明确拆开

开源整理最容易出问题的地方，不是功能，而是边界。

本项目将以下内容明确隔离：

- 公开分支可见：README、快速启动、通用部署、案例墙、规则入口
- 内部资料隔离：`docs/internal/**`
- 发布模型分离：`develop` 负责日常开发，`master` 负责公开发布

这样 AI 在整理仓库、补文档、发布代码时，有明确的公开边界可遵守。

### 3. 把验证命令写清楚，避免“看起来改好了”

本项目要求文档、规则、代码改动后都给出实际执行过的验证命令和结果。

这比“AI 说已经改好”重要得多，因为真正可用的协作开发一定要有可验证闭环。

## 如果你想把这套方式迁移到自己的项目

建议按这个顺序做：

1. 先补一个仓库级 `AGENTS.md`
2. 再拆出 `rules/`，写清接口、分页、目录边界和验证要求
3. 再补 `skills/` 或同类任务工作流说明
4. 最后再让 AI 参与 README、文档和代码整理

顺序不要反过来。否则 AI 只会快速产出很多内容，但不会稳定地产出正确内容。

## 给开源项目维护者的建议

如果你的目标是长期涨 Star、降低协作成本、提高二开成功率，那么优先级应该是：

1. 先把仓库边界写清楚
2. 再把验证方式写清楚
3. 再把 AI 协作资产公开出来
4. 最后才是做内容分发

没有前面三步，外部传播只会把更多用户带到一个还没整理好的仓库首页。
